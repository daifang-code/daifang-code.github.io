缺陷描述

C++的拷贝赋值操作符没有验证自我赋值问题。例如下面代码：

class Foo  
{  
private：  
    string *str;  
    int val;  
public:  
    ...  
    Foo & operator=(const Foo &rhs)  //一份不安全的operator=实现版本  
    {  
        delete str;  
        str = new string(*(rhs.str));  
        val = rhs.val;  
        return *this;  
    }  
}

这里的自我赋值问题是，operator=函数内的*this和rhs有可能是同一个对象，如果他们是同一个对象，delete就不仅销毁是(*this).str中所指向的对象，他也销毁rhs.str中所指向的对象。当执行第二行时，构造的是一个空的str，因为当前对象已经被销毁了。这就会导致使用已经释放的内存的问题。

漏洞与风险

使用已经释放的内存

operator=函数内的*this和rhs有可能是同一个对象，如果他们是同一个对象，delete就不仅销毁是(*this).str中所指向的对象，他也销毁rhs.str中所指向的对象。当执行第二行时，构造的是一个空的str，因为当前对象已经被销毁了。这就会导致使用已经释放的内存的问题。

漏洞利用威胁

低

缓解与预防

在拷贝赋值运算符中必须处理自赋值情况。

class good {

  int y;
public:
  good(int xx, int yy) : x(xx), y(yy) {}
  good(const point &init) : x(init.x), y(init.y) {}

    return *this;

函数参数过大值传递

在 2024-九月-28 发布于 代码风格和效率 分类 • 1 min read

  • 缺陷描述
  • 漏洞与风险

缺陷描述

在C和C++程序中，如果函数的参数是通过值传递的，那在将实参赋值给形参过程中会在内存中临时复制该参数的值，从来用来进行函数内的相应操作。如果该参数过大，则会影响运行效率。

漏洞与风险

如果传递参数为结构体或类，将会严重的影响效率，特别当该函数被多次调用时。

漏洞利用威胁

低

缓解与预防

如果传递的实参大于16字节时，特别是传递参数为数组，结构体，或类时，采用指针传递参数。

演示实例

    int a[20];
    int b[20];
    int c[20];

}

负整数误用

在 2024-九月-28 发布于 代码风格和效率 分类 • 1 min read

  • 缺陷描述
  • 漏洞与风险
  • 漏洞利用威胁

  • 演示实例

缺陷描述

负整数和函数返回值（可能为负）必须在使用前进行检查（例如，作为数组索引，循环边界，代数表达式变量或系统调用的大小/长度参数）。

常见的负整数误用包括： •在将其用作静态数组索引之前，将负值分配给带符号的整数变量。 •直接使用负函数或通过将其强制转换为无符号整数来返回值。 隐式转换为无符号整数的有符号负整数将产生非常大的值。如果在使用该值之前对其进行了错误的边界检查，则进程可能会分配过多的内存，允许循环处理的时间过长，内存溢出和损坏或产生整数溢出。 这些缺陷很难被发现，因为它们的影响可能不会在流程执行过程中立即出现。

漏洞与风险

负整数误用会导致内存损坏，进程崩溃，无限循环，整数溢出和安全缺陷（如果用户能够控制未经检查的输入）。

漏洞利用威胁

严重
