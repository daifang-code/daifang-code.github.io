谨防零长度分配内存

在 2024-九月-28 发布于 内存和资源误用 分类 • 1 min read

缺陷描述

当请求分配内存的大小为0时，内存分配函数malloc()，calloc()和realloc()的行为是不可控的。C标准[ISO/IEC 9899：2011]的条款7.22.3规定：

  • 如果请求的空间大小为零，则行为是由编译器实现定义的：或者返回空指针，或者返回的指针不能用于访问。

当内存分配函数返回非空指针的情况下，读取或写入分配的内存区域会导致未定义的行为。

漏洞与风险

读取或写入分配0字节长度的内存可能会导致程序异常终止。

缓解与预防

谨防零长度分配内存。

漏洞利用威胁

高

主机字节顺序与网络字节顺序的错误转换

在 2024-九月-28 发布于 数值计算错误 分类 • 1 min read

缺陷描述

来源于主机边界的多字节的值在发送或者写入到环境之前，未将主机字节顺序转换为网路字节顺序或从外界获取或者读取一个多字节到主机边界时，没有把网络字节顺序转换为主机字节顺序。

漏洞与风险

程序在系统下运行伴随不同的字节顺序方法，这些方法需相互之间连通。错误地转换字节顺序可能导致程序功能实现不正确。

漏洞利用威胁

低

缓解与预防

应使用htons函数把unsigned short类型从主机序转换到网络序；应使用htonl 函数把unsigned long类型从主机序转换到网络序；应使用ntohs 函数把unsigned short类型从网络序转换到主机序；应使用ntohl 函数把unsigned long类型从网络序转换到主机序。

缺陷描述

程序在使用后没有正确地跟踪和释放分配的内存，这会缓慢消耗剩余的内存。

漏洞与风险

不正确地释放内存(内存泄漏)

程序在使用后没有正确地跟踪和释放分配的内存，这会缓慢消耗剩余的内存。

这通常是由于错误处理格式错误的数据或者意外中断会话而引发的。

漏洞利用威胁

中等

缓解与预防

阶段: 实施

策略: 库或框架

选择一种提供自动内存管理的语言或工具，或者使手动内存管理不易出错。 例如，Linux中的glibc提供了对于无效指针错误的保护。

当使用Xcode定位OS X或iOS时，启用自动引用计数（ARC）[ REF-391 ]。

为了在进行C++编程时正确和一致地管理内存，可考虑使用智能指针类，如std :: auto_ptr（由ISO / IEC ISO / IEC 14882：2003定义），std :: shared_ptr和std :: unique_ptr（由即将发布的C++标准修订版，非正式地称为C++ 1x）或等效解决方案（如Boost）。

阶段: 架构和设计

使用抽象库来抽象化有风险的API，这并不是一个完整的解决方案。

阶段: 架构与设计; 构建与编译

使用Boehm-Demers-Weiser垃圾收集器或valgrind来检测代码中的泄漏。

注意：这并不是一个完整的解决方案，因为它不是100％有效。

在 2024-九月-28 发布于 内存和资源误用 分类 • 1 min read

缺陷描述

在C和C++语言中，字符串通常以'\0'结尾。求解字符串的长度时，通常是从字符串起始位置统计字符个数直到找到'\0'。 很多以字符串作为参数的函数，都会要求该字符串带有null终结符。传入不带有null终结符的字符序列作为参数给该类函数，会导致对该对象外的内存的访问。

漏洞与风险

越界访问内存

很多以字符串作为参数的函数，都会要求该字符串带有null终结符。传入不带有null终结符的字符序列作为参数给该类函数，会导致对该对象外的内存的访问。

漏洞利用威胁

中

缓解与预防

在使用字符串之前，保证字符串以'\0'结尾。

缺陷描述

用户命令行输入可能很长，导致数组越界。

漏洞与风险

在存储器缓冲区范围内的不当操作限制

该软件在内存缓冲区上执行操作，但它可以读取或写入缓冲区预期边界之外的内存位置。 某些语言允许直接寻址内存位置，并且不会自动确保这些位置对于正在引用的内存缓冲区有效。这可能导致对可能与其他变量，数据结构或内部程序数据相关联的内存位置执行读取或写入操作。 因此，攻击者可能能够执行任意代码，改变预期的控制流程，读取敏感信息或导致系统崩溃。

漏洞利用威胁

高

缓解与预防

阶段：架构和设计

使用经过审查的库或框架，不允许发生这种弱点或提供使这种弱点更容易避免的构造。 包括Messier和Viega的安全C字符串库（SafeStr）[REF-57]和Microsoft的Strsafe.h库[REF-56]。这些库提供了更安全的溢出倾向字符串处理函数版本。 注意：这不是一个完整的解决方案，因为许多缓冲区溢出与字符串无关。

阶段：构建和编译

使用自动提供缓解或消除缓冲区溢出的保护机制的功能或扩展来运行或编译软件。 例如，某些编译器和扩展提供了内置于编译代码中的自动缓冲区溢出检测机制。示例包括Microsoft Visual Studio / GS标志，Fedora / Red Hat FORTIFY_SOURCE GCC标志，StackGuard和ProPolice。 注意：这不一定是一个完整的解决方案，因为这些机制只能检测某些类型的溢出。另外，攻击仍可能导致拒绝服务，因为典型的响应是退出应用程序。

阶段：实施

分配和管理应用程序内存时，请考虑遵守以下规则： 仔细检查您的缓冲区是否与您指定的一样大。 使用接受许多字节进行复制的函数(如strncpy())时，请注意，如果目标缓冲区大小等于源缓冲区大小，则可能不会以NULL结束字符串。 如果访问循环中的缓冲区，请检查缓冲区边界，并确保您没有写入已分配空间的危险。 如果需要，在将所有输入字符串传递给复制和串联函数之前，将其截断为合理的长度。

阶段：操作

使用随机排列程序可执行程序和库的位置的功能或扩展来运行或编译软件。因为这会使地址不可预知，所以它可以防止攻击者可靠地跳转到可利用的代码。 例如包括地址空间布局随机化（ASLR）[REF-58] [REF-60]和位置无关可执行文件（PIE）[REF-64]。 注意：这不是一个完整的解决方案。但是，它会迫使攻击者猜测一个未知值，这会改变每个程序的执行。另外，攻击仍可能导致拒绝服务，因为典型的响应是退出应用程序。

阶段：操作

使用提供数据执行保护（NX）或其同等功能[REF-60] [REF-61]的CPU和操作系统。 注意：这不是一个完整的解决方案，因为缓冲区溢出可以用来覆盖附近的变量以危险的方式修改软件的状态。另外，它不能用于需要自修改代码的情况。最后，攻击仍然会导致拒绝服务，因为典型的响应是退出应用程序。

阶段：实施

将无界复制函数替换为支持长度参数的类似函数，如strcpy和strncpy。如果它们不可用，则创建它们。 注意：这种方法仍然容易受到计算错误的影响，包括诸如错误的错误（CWE-193）和错误地计算缓冲区长度（CWE-131）等问题。

    value = array[index];
    }
    // if array index is invalid then output error message
    // and return value indicating error
    else {
        printf("Value is: %d\n", array[index]);
        value = -1;
    }
    return value;
}

在以上代码中，方法从特定数组索引位置的数组中检索一个值，该值为方法的输入参数。 但是，此方法仅验证给定的数组索引是否小于数组的最大长度，但不检查最小值（CWE-839）。这将允许一个负值被接受为输入数组索引，这将导致超出边界的读取（CWE-125）并且可能允许访问敏感内存。应检查输入数组索引以验证是否在数组所需的最大和最小范围内（CWE-129）。在这个例子中，if语句应该修改为包含最小范围检查，如下所示。

缺陷描述

缺陷描述

移位的数量不能是负数或大于等于提升后左操作数的位数。一个整型的精度是它表示数值的位数，包含符号位和填充位。对于无符号整型来说，它的宽度和精度是一样的。然而对于有符号整数来说，它的宽度大于它的精度。这条规则使用精度代替宽度来表示，因为几乎在每一种情况下，在移位操作中移位的数量大于或等于操作数的精度时会导致一个错误（逻辑错误）。不同于溢出，逻辑错误是一种表示形式方面的缺陷。

缺陷描述

不同的数据类型在转换时会造成精度丢失的情况。(MISRA)-C:2004 对类型转换做了如下约束： MISRA-C 10.1 整型表达式如果存在以下情况，则不应该隐式转换为一个不同的类型:

  1. 不是转换为一个同符号的宽类型；
  2. 整型表达式是一个合成表达式；
  3. 整型表达式不是一个常量，而是函数的参数；
  4. 整型表达式不是一个常量，而是一个返回值。

MISRA-C 10.2. 浮点型表达式如果存在以下情况，则不应该隐式转换为一个不同的类型:

  1. 不是转换为一个同符号的宽类型；
  2. 浮点型表达式是一个合成表达式；
  3. 浮点型表达式是函数的参数；
  4. 浮点型表达式是一个返回值。

MISRA-C 10.3. 整型合成表达式只能转换为同符号同类型并且位数更大的值。

MISRA-C 10.4. 浮点型合成表达式只能转换为同符号同类型并且位数更大的值。

MISRA-C 10.5. 对unsigned char 或 unsigned short类型的变量使用位操作~或<<时，必须立即将结果转换为unsigned char 或 unsigned short类型。

缺陷描述

如果一个类被其他类继承，应该使用虚函数做析构函数。例如，如果B继承A，我们用A* x = new B; 然后delete x, 这时应该调用B的析构函数，但是如果A的析构函数不是virtual, 则不会调用B的析构函数，这样B维护的内存可能会泄漏

漏洞与风险

一般情况下类的析构函数里面都是释放内存资源，而如果基类的析构函数不是虚函数，那个继承类的析构函数就不被调用，这样的话就会造成内存泄漏。

缓解与预防

如果一个类作为基类被其他类继承，那么该类的析构函数应该用虚函数

描述

使用已释放的内存或重复释放已经被释放的内存，根据缺陷的具体实例化和时序关系，可能导致程序崩溃或任意代码执行等多种不良后果。最常见的情况是系统使用了被释放的内存导致崩溃。使用被释放的内存有两种常见原因： 对释放的错误判断及其他异常情况； 程序中哪一部分负责释放内存定义不清；

在这种情况下，被释放的指针在某些时候会处在另一个有效指针的作用范围内。原始的已经被释放的指针再次被使用进行数据更改时，会导致有效的内存数据被破坏，这将在程序中引起不确定的行为。

如果新分配的指针拥有一个对象。例如在C++中，不同的函数指针被分散在堆内存中。如果其中一个函数指针被shellcode的地址覆盖，就会导致任意代码执行漏洞。

风险评估

  • 如果有问题的内存区域在其它地方被正确分配和使用，则使用先前释放的内存可能会破坏其中的有效数据。
  • 如果在内存释放之后发生了数据合并，则无效数据被用作内存块信息时，该过程可能会崩溃。
  • 如果在内存块合并发生前输入了恶意数据，则可能被用来执行任意代码。

漏洞利用威胁

严重

缓解与预防

  • 确保内存不再使用后再释放。
  • 在释放一个指针后，将其置为空指针。但是，使用多个或复杂的数据结构可能会降低该策略的实用性。

在 2024-九月-28 发布于 类型和封装 分类 • 1 min read

  • 缓解与预防

Summary：类的成员函数为const和static函数

缺陷描述

类成员函数为const函数，虽然函数可以从技术上得到满足，而且这个函数不会显示编译错误，但从概念上讲是没有意义的。成员函数可以做一个静态函数。使一个函数静态可以带来性能好处，因为没有将实例传递给函数。这种更改不会导致编译器错误，但在概念上它不一定是有意义的。

在资源使用周期内的不当控制（问题1、2、3、4、5）

在创建，使用和发布的整个生命周期中，软件没有维护或不正确地维护对资源的控制权。

资源往往有明确的说明如何创建，使用和销毁。当软件不遵循这些说明时，它可能会导致意外行为和潜在的可利用状态。

即使没有明确的指示，也应遵守各种原则，例如“在创建完成之前不要使用对象”，或者“在对象被销毁后不要使用它”。

使用过期的文件描述符（问题1）

该软件在关闭后使用或访问文件描述符。

在特定文件或设备的文件描述符被释放后，它可以被重新使用。该代码可能不会写入原始文件，因为重用的文件描述符可能会引用不同的文件或设备。

在存储器缓冲区范围内操作的不当限制（问题6）

该软件在内存缓冲区上执行操作，但它可以读取或写入缓冲区预期边界之外的内存位置。

某些语言允许直接寻址内存位置，并且不会自动确保这些位置对于正在引用的内存缓冲区有效。这可能导致对可能与其他变量，数据结构或内部程序数据相关联的内存位置执行读取或写入操作。

因此，攻击者可能能够执行任意代码，改变预期的控制流程，读取敏感信息或导致系统崩溃。

函数调用的参数数量不正确（问题7）

在 2024-九月-28 发布于 API误用 分类 • 1 min read

  • 缺陷描述

    • 缓冲区溢出

缺陷描述

函数原型： extern char *strncat(char *dest,char *src,int n)
参数说明：src为源字符串，dest为目的字符串，n为指定的src中的前n个字符。
函数功能：把src所指字符串的前n个字符添加到dest结尾处，覆盖dest结尾处的'\0'，实现字符串连接。

strncat()的最后一个参数不应该是缓冲区的总长度，而应该是调用strncpy()后缓冲区剩余的长度。这两个函数都要求指定剩余的长度而不是缓冲区的总长度。由于剩余的长度在每次添加或删除数据时都会改变，因此程序员必须跟踪这些改变或重新计算剩余长度。这个过程很容易出错，并且可能会导致缓冲区溢出。
下面的调用在使用strncat()连接字符串之前正确地计算了剩余的空间： 

1
strncat(dest, src, sizeof(dest) - strlen(dest) - 1);

漏洞与风险

缓冲区溢出

错误地计算n的值可能会导致拼接到dest的字符串超出dest的内存大小，导致缓冲区溢出。

漏洞利用威胁

中

正确地计算需要拷贝的长度。

  // Replace with 'sizeof(dest) - strlen(dest) - 1' or use a safer 'strlcat' API}}

缺陷描述

声明可变长度数组时，数组长度必须满足以下要求：

  1. 数组长度不能为负数或0。
  2. 数组长度必须是一个初始化的值。
  3. 数组长度不能为一个污点数据。

漏洞与风险

意料之外的结果

声明可变长度数组时，如果不满足以上要求，可能申请的数组与预期不符，产生意料之外的结果。

漏洞利用威胁

低

缓解与预防

声明可变长度数组时，数组长度必须满足以下要求：

  1. 数组长度不能为负数或0。
  2. 数组长度必须是一个初始化的值。
  3. 数组长度不能为一个污点数据。

缺陷描述

VA_LIST的用法：

  1. 首先在函数里定义一具VA_LIST型的变量，这个变量是指向参数的指针；
  2. 然后用VA_START宏初始化变量刚定义的VA_LIST变量；
  3. 然后用VA_ARG返回可变的参数，VA_ARG的第二个参数是你要返回的参数的类型（如果函数有多个可变参数的，依次调用VA_ARG获取各个参数）；
  4. 最后用VA_END宏结束可变参数的获取。

错误的使用VA_LIST，可能会导致产生错误的结果。

漏洞与风险

意料之外的结果

错误的使用VA_LIST，可能会导致产生错误的结果。

缺陷描述

所有的针对互斥量的加锁解锁操作，都必须针对同一模块，并且在同一抽象层面进行。否则，将会可能导致某些加锁/解锁操作不会依照多线程设计而被执行，甚至依照锁的类型，最终导致死锁、资源竞争等其他漏洞的爆发。在对加锁对象进行解锁过程中，错误的解锁顺序，容易造成死锁。

漏洞与风险

死锁

不正确地加锁解锁操作，导致死锁、资源竞争等问题。

漏洞利用威胁

低

缓解与预防

正确地执行加锁解锁操作。对于死锁问题，可以采取例如可以调整资源请求顺序，或者加上sleep操作等方式。

缺陷描述

在代码中使用memset函数来重置内存的值会与期望获取的大小不符。这种情况通常发生在sizeof关键字的使用上面。意图使用sizeof计算实际类型的大小，结果却计算了指针的大小。这种情况会导致重置内存少于期望重置的大小，最后导致未定义行为。

漏洞与风险

未定义行为

使用sizeof计算实际类型的大小，结果却计算了指针的大小，可能会导致未定义的行为。

漏洞利用威胁

低

缓解与预防

正确地使用sizeof操作符。

    char name[128];
};
void test() {
    struct BB buf;
    memset(&buf, 0, sizeof(&buf)); // bad

  • 缺陷描述

缺陷描述

由于浮点数可以表示实数，因此程序员常常错误地认为它们可以准确地表示任何简单的小数。浮点数也和整数一样受到精度的限制，二进制浮点数无法准确地表示所有的十进制小数，即使后者只用几个十进制数字就可以表示。

不同的实现具有不同的精度限制，为了使代码保持可移植性，不应该使用浮点数作为循环计数器。

漏洞与风险

死循环

使用浮点数作为循环计数器，可能会导致循环无法终止，造成死循环。

漏洞利用威胁

缺陷描述

满足如下条件时触发数据竞争

  • 一个进程中两个或多个线程同时访问同一个内存区域；

  • 这些线程没有使用锁去控制对该内存区域的访问。

漏洞与风险

不可预测的行为

当数据竞争发生时，访问的顺序是不确定的，并且不同的运行依据不同的执行顺序，会给出不同的计算结果。虽然有些数据竞争是良性的（例如循环等待中的内存访问），但是很多数据竞争在程序中是缺陷。

漏洞利用威胁

低

缺陷描述

C和C++没有提供内置的防护措施来防止在任意内存中访问或覆盖数据，也没有自动检测写入数组的数据是否在数组的边界以内。

缓冲区溢出的后果包括有效的数据被覆盖、任意潜在的恶意代码执行。

以下字符串拷贝函数可能导致缓冲区溢出：

  • strcpy, strcat
  • wcscpy, wcscat
  • StrCpy, StrCpyA, StrCpyW, StrCat, StrCatA, StrCatW
  • OemToChar, OemToCharA, OemToCharW, OemToAnsi, OemToAnsiA, OemToAnsiW
  • _mbscpy, _mbscat, _tcscat, _tcscpy
  • lstrcpy, lstrcpyA, lstrcpyW,
  • lstrcat, lstrcatA, lstrcatW

漏洞与风险

缓冲区溢出

缓冲区溢出的后果包括有效的数据被覆盖、任意潜在的恶意代码执行。

高

缺陷描述

有限的资源包括内存，文件系统存储，数据库连接池条目和CPU。如果攻击者可以触发这些有限资源的分配，但是资源的数量或大小不受控制，则攻击者可能消耗所有可用资源，导致拒绝服务。这将阻止正常用户访问该软件，并且可能对软件执行环境产生影响。例如，对应用程序的内存耗尽攻击可能会降低应用程序及其主机操作系统的速度。

至少有三种不同的情况通常会导致资源耗尽： 缺少分配的资源数量的限制 在关闭之前，丢失对资源的所有引用 * 处理后不关闭/返还资源

  • 资源耗尽的最常见结果是拒绝服务。该软件可能会变慢，由于未处理的错误而崩溃或将合法用户锁定
  • 在某些情况下，资源耗尽会导致软件“打开失败”。 软件的状态以及安全功能可能会被损坏。

漏洞与风险

使用释放过的内存

显式地释放一个接口指针并没有修改指针的引用计数，这样会导致其它拥有这个指针权限的接口继续使用该释放的指针，可能会造成使用释放内存的缺陷。

缺陷描述

由C++编译器生成的默认的拷贝构造函数和赋值操作符只能实现类的数据成员的浅复制。因此，如果没有显式定义拷贝构造函数和赋值操作符，可能产生内存泄漏、资源泄漏、使用已释放的内存、重复释放内存等错误。

低

缓解与预防

计算表达式不清晰

在 2024-九月-28 发布于 逻辑错误 分类 • 1 min read

缺陷描述

计算表达式表述不清楚，使得同一个表达式有多种理解方式。例如'a+b?c:d'可能原意是想表达(a+b)?c:d或a+(b?c:d)。当一个if语句被分号提前终止时，then语句将永远被执行，从而改变了if语句的逻辑。在while,for等循环语句被分号提前终止时，可能会导致死循环或者循环体部分只被执行一次。

漏洞与风险

若表达式没有被正确理解，或者程序执行逻辑与原意不同，则整个程序将脱离预定轨迹运行，甚至形成死循环并导致程序崩溃。

缓解与预防

对于表达式，尽量多用括号标明运算步骤。对于判断和循环语句，注意分号所在位置。
