缺陷描述

operator=()没有对所有成员变量赋值

operator= 不符合c/c++规范，operator=应该返回一个对自身的引用（使用return *this）

operator=用在相同变量（a=a）是危险的，可能造成动态内存的错误，请检查确定这里安全

漏洞与风险

对自己赋值除了效率低下以外，一个赋值运算符必须首先释放掉一个 对象的资源（去掉旧值），然后根据新值分配新的资源。在自己给自己赋值的情 况下，释放旧的资源将是灾难性的，因为在分配新的资源时会需要旧的资源。

漏洞利用威胁

中

缓解与预防

让 operator=返回*this 的引用

在 operator=中对所有数据成员赋值

在 operator=中检查给自己赋值的情况

演示实例

漏洞代码示例

Derived& Derived::operator=(const Derived& rhs)
{
  if (this == &rhs) return *this;
  y = rhs.y; // 给 Derived 仅有的数据成员赋值
  return *this;
}

在上面的代码中，Derived 对象的 Base 部分的数据成员 x 在赋值运算符中未受影响，调用operator=，变量的Base部分不会被赋值操作改变。

Derived& Derived::operator=(const Derived& rhs)
{
  if (this == &rhs) return *this;
  Base::operator=(rhs); // 调用 this->Base::operator=
  y = rhs.y;
  return *this;
}

在上面的代码中，在 Derived 的赋值运算符里显式地对 Derived 的 Base 部分赋值。

缺陷描述

函数传递的实参类型与形参不一致

漏洞与风险

如果实参和形参不一致，则最终调用以形参为准。这样会在调用函数时改变参数的类型，从而导致结果的差异。

缓解与预防

使形参和实参类型一致，如果不能一致则在编写时考虑形参的类型。

漏洞利用威胁

高

演示示例

漏洞缺陷示例

1
2
3
void bad(int a);
char b=30; //detect
bad(b)

b是实参，它与形参的类型是不一样的。在调用时系统将默认b为int型并带入函数中。但是这样会导致b类型的改变

漏洞缺陷修复示例

1
2
3
void bad(int a);
int b=30;
bad(b)

缺陷描述

初始化列表成员顺序不正确可能导致部分成员未正确初始化。

漏洞与风险

未正确初始化

成员变量的初始化顺序由内存中的排列顺序决定。如果对象A的初始化依赖于对象B，而对象A比对象B先初始化，这回导致初始化不正确。

漏洞利用威胁

低

缓解与预防

保证正确的初始化顺序。

演示实例

漏洞代码示例

{
  private:
    int a,b;
  public:
    A():b(1),a(b+1){}
    ...
}

在上面的代码中，初始化的顺序为a,b，但a的初始化依赖于b。

漏洞代码修复示例

1
2
3
4
5

8

{
  private:
    int b,a;
  public:
    A():b(1),a(b+1){}
    ...
}

在上面的代码中，保证了正确的初始化顺序。

缺陷描述

把指针赋值给整型变量或者把整型变量赋值给指针等不当的数值处理。

漏洞与风险

把指针值赋值给整型变量，可能在后续代码由于整形运算，比如乘法，得出无效的指针值。 把整形变量赋值给指针，可能在后续代码引起使用没有声明定义，或者无法预期的内存地址。

漏洞利用威胁

低

缓解与预防

保证整型变量和指针之间不进行互相赋值。

演示实例

漏洞代码示例

1
2
3
4
5
6
int bad(int *p)
{
    int a = p;
    int b = a + 4;
    return b;
}

指针p赋值给整型a。

漏洞代码修复示例

1
2
3
4
int* foo(int *p)
{
    return p + 4;
}

在上面的代码中，保证指针p不复制给其他整型。

结构体填充导致内存浪费

在 2024-九月-28 发布于 类型和封装 分类 • 1 min read

  • 缺陷描述
  • 漏洞与风险
    • 资源浪费
  • 漏洞利用威胁
  • 缓解与预防
  • 演示实例
    • 漏洞代码示例
    • 漏洞代码修复示例

缺陷描述

结构体成员根据它们在程序文本中的声明顺序排列。结构体中可能会使用填充字符，保证结构在内存中正确对齐。结构体填充是为了在多重的架构上方便成员快速访问。
如果结构体中的成员定义顺序不当，可能会造成过多的结构体填充的产生，导致系统内存的浪费。

漏洞与风险

资源浪费

如果结构体中的成员定义顺序不当，可能会造成过多的结构体填充的产生，导致系统内存的浪费。

漏洞利用威胁

低

缓解与预防

检查结构体成员的放置顺序，重排结构体成员节省不必要的空间使用。

演示实例

漏洞代码示例

1
2
3
4
5
struct bad {
  char c;
  int i;
  char c1;
};

在上面的代码中，由于结构体的成员分布不当，其中每个字符类型（char）的成员，都要占用一个独立的int位长，造成了过多的结构体填充的产生。

漏洞代码修复示例

1
2
3
4
5
struct bad {
  char c;
  char c1;
  int i;
};

在上面的代码中，两个char类型的成员定义相连，共占用一个int位长，减少了过多的结构体填充造成的内存浪费。

不兼容的指针类型转换

在 2024-九月-28 发布于 类型和封装 分类 • 1 min read

  • 缺陷描述
  • 漏洞与风险
    • 越界访问内存
  • 漏洞利用威胁
  • 缓解与预防
  • 演示实例
    • 漏洞代码示例
    • 漏洞代码修复示例

缺陷描述

将一种指针类型转换为不兼容的指针类型时，可能会导致越界访问内存隐患。例如将一个指向小数据类型的指针转换为一个指向大数据类型的指针时，通过转换后的指针访问内存对象时，可能会导致越界访问内存。

漏洞与风险

越界访问内存

将一个指向小数据类型的指针转换为一个指向大数据类型的指针时，通过转换后的指针访问内存对象时，可能会导致越界访问内存。

漏洞利用威胁

中

缓解与预防

在做指针类型转换时，将指针转换为一种兼容类型。

演示实例

漏洞代码示例

1
2
3
4
5
6
void init(int *x) { *x = 42; }

void foo() {
  uint16_t y;
  init((int *)&y); 
}

在第5行时，将uint16_t* 类型转为int* 类型，uint16_t* 类型指向的内存块占16个字节，int* 类型指向的内存块占32个字节，在第1行中以32个字节的大小给该内存块(大小为16字节)赋值，就会造成越界访问内存。

漏洞代码修复示例

1
2
3
4
5
6
void init(uint16_t *x) { *x = 42; }

void foo() {
  uint16_t y;
  init((uint16_t *)&y); 
}

在第5行时，做兼容的指针类型转换。

类型转换导致精度丢失

在 2024-九月-28 发布于 类型和封装 分类 • 1 min read

  • 缺陷描述
  • 漏洞与风险
    • 精度丢失
  • 漏洞利用威胁
  • 缓解与预防
  • 演示实例
    • 漏洞代码示例
    • 漏洞代码修复示例

缺陷描述

不同的数据类型在转换时会造成精度丢失的情况。(MISRA)-C:2004 对类型转换做了如下约束： MISRA-C 10.1 整型表达式如果存在以下情况，则不应该隐式转换为一个不同的类型:

  1. 不是转换为一个同符号的宽类型；
  2. 整型表达式是一个合成表达式；
  3. 整型表达式不是一个常量，而是函数的参数；
  4. 整型表达式不是一个常量，而是一个返回值。

MISRA-C 10.2. 浮点型表达式如果存在以下情况，则不应该隐式转换为一个不同的类型:

  1. 不是转换为一个同符号的宽类型；
  2. 浮点型表达式是一个合成表达式；
  3. 浮点型表达式是函数的参数；
  4. 浮点型表达式是一个返回值。

MISRA-C 10.3. 整型合成表达式只能转换为同符号同类型并且位数更大的值。

MISRA-C 10.4. 浮点型合成表达式只能转换为同符号同类型并且位数更大的值。

MISRA-C 10.5. 对unsigned char 或 unsigned short类型的变量使用位操作~或<<时，必须立即将结果转换为unsigned char 或 unsigned short类型。

漏洞与风险

精度丢失

不同的数据类型在转换时会造成精度丢失的情况，精度丢失可能导致程序功能不正确。

漏洞利用威胁

中

缓解与预防

检查转换前和转换后的类型，预防精度丢失。

演示实例

漏洞代码示例

1
2

typedef double float64_t;

const int32_t xs32a = 0, xs32b = 1;
static void bad() {
  /* Defect: Casting complex expression with
  integer type to a non-integer
  type float64_t */
  (void)(float64_t)(xs32a / xs32b);
}

在上面的代码中，将整型表达式xs32a / xs32b显式转换为float64_t类型，违反了MISRA-C 10.3。

}

在上面的代码中，将整型表达式xs32a / xs32b显式转换为int32_t类型，符合MISRA-C 10.2。

软件调用的函数过程或者例程的过程中，调用者指定的参数是错误的数据类型。这种缺陷最可能发生在松散类型的语言中，或者在编译时无法强制实现可变参数类型的强类型语言中，或者存在隐式强制类型转换的语言中。 例如不合法的scanf函数或print函数参数的数量，位置问题。string，int，unsigned int，signed int，float类型不匹配等问题。

不正确的类型转换

软件不能将对象，资源或结构正确地从一种类型转换为一种不同的类型。

使用错误的指定的参数值的函数调用

软件调用一个函数，过程或者例程，但调用者指定一个包含错误值的参数。

没有使用虚函数作为析构函数

在 2024-九月-28 发布于 类型和封装 分类 • 1 min read

  • 缺陷描述
  • 漏洞与风险
  • 缓解与预防
  • 漏洞利用威胁
  • 演示示例

缺陷描述

如果一个类被其他类继承，应该使用虚函数做析构函数。例如，如果B继承A，我们用A* x = new B; 然后delete x, 这时应该调用B的析构函数，但是如果A的析构函数不是virtual, 则不会调用B的析构函数，这样B维护的内存可能会泄漏

漏洞与风险

一般情况下类的析构函数里面都是释放内存资源，而如果基类的析构函数不是虚函数，那个继承类的析构函数就不被调用，这样的话就会造成内存泄漏。

缓解与预防

如果一个类作为基类被其他类继承，那么该类的析构函数应该用虚函数

缺陷描述

一个类的构造函数通常需要初始化类中所有的非静态成员变量。这是一种常见的代码标准。未初始化成员变量是一种不安全的行为，因为使用未初始化的变量会导致意料之外的情况发生。

漏洞与风险

意料之外的行为

由于分配给变量的内存可能在之前某个程序或函数使用过，因此使用未初始化的变量可能产生意想不到的异常行为和错误结果。

在构造函数中初始化所有非静态的成员变量。

缺陷描述

在用typedef定义structure或union的别名时使用packed属性。

漏洞与风险

packed属性只能用于structure或union的类型定义。packed属性用于typedef中会被忽略。
"packed"的作用是使字节对齐，一些平台对某些特定类型的数据只能从某些特定地址开始存取，如一个变量没有进行字节对齐，有些架构的CPU在访问时可能会发生错误；如果不按照适合其平台要求对数据存放进行对齐，会降低存取的效率。

漏洞利用威胁

低

缓解与预防

不在用typedef定义structure或union的别名时使用packed属性。

演示实例

漏洞代码示例

2
3
4

{
    ...
} __attribute__((packed)) S;  // packed属性将会忽略

漏洞代码修复示例

1
2
3
4
typedef struct    
{
    ...
} S;

缺陷描述

指针类型和非指针类型间进行类型转换。

漏洞与风险

使用时依赖平台或体系结构，指针可能会也可能不会由相同的比特数作为一个整数类型来表示，如无符号整型，一般情况下，指针类型和非指针类型之间的转换认为是不安全的。

漏洞利用威胁

低

缓解与预防

不宜在一个整型中存储指针值。如果指针指向的类型必须被隐藏，使用void类型代替。

{
  char* a = "hello world";
  unsigned int b = (unsigned int)a;  //指针类型和非指针类型间进行类型转换
  ...
}

在上面的代码中，将指针变量a转换为unsigned int类型。

类只定义了拷贝构造函数或只定义了赋值操作符

缺陷描述

类只定义了拷贝构造函数而没有定义赋值操作符，或只定义了赋值操作符而没有定义拷贝构造函数。

漏洞与风险

由C++编译器生成的默认的拷贝构造函数和赋值操作符只能实现类的数据成员的浅复制。因此，如果没有显式定义拷贝构造函数和赋值操作符，可能产生内存泄漏、资源泄漏、使用已释放的内存、重复释放内存等错误。

漏洞利用威胁

低

缓解与预防

若类的成员变量包含动态分配的内存或资源，应显式地定义拷贝构造函数和赋值操作符。

演示实例

漏洞代码示例

1
2
3
4
5
class MyClass  //类只定义了拷贝构造函数
{
public:
  MyClass(const MyClass &a);  //拷贝构造函数
};

在上面的代码中，类MyClass只定义了拷贝构造函数，缺少赋值操作符。

漏洞代码修复示例

1
2
3
4
5
6
class MyClass      
{
public:
  MyClass(const MyClass &a);  ///拷贝构造函数
  MyClass &operator=(const MyClass &a);  //赋值操作符
}
