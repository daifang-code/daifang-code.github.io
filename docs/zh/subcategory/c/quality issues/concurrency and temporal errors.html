不正确地使用vfork

在 2024-九月-28 发布于 并发和时序错误 分类 • 1 min read

  • 漏洞利用威胁
  • 缓解与预防
  • 演示实例
    • 漏洞代码示例

缺陷描述

如果vfork()函数创建的进程修改了除了pid_t类型以外的变量，或者从调用vfork()的函数返回，或者在成功调用_exit()函数或exec()一类函数前调用了其他函数，会导致未定义行为的发生。

漏洞与风险

意料之外的结果

如果vfork()函数创建的进程修改了除了pid_t类型以外的变量，或者从调用vfork()的函数返回，或者在成功调用_exit()函数或exec()一类函数前调用了其他函数，会导致未定义行为的发生。

漏洞利用威胁

低

缓解与预防

避免在在成功调用_exit()函数或exec()一类函数前修改任何程序变量。

演示实例

漏洞代码示例

2
3
4
void bad(int x) {
  pid_t pid = vfork();
  x = 0;
}

在上面的代码中，在调用了vfork()函数后，对类型为int的变量x进行修改，该行为会导致未定义行为的发生。

漏洞代码修复示例

1
2
3
4
void good(int x) {
  pid_t pid = vfork();
  pid = 0;
}

在上面的代码中，在调用了vfork()函数后，修改的变量pid的类型为pid_t，该行为不会产生未定义行为。

加解锁错误

在 2024-九月-28 发布于 并发和时序错误 分类 • 1 min read

  • 演示实例

    • 漏洞代码修复示例1

    • 漏洞代码修复示例2

缺陷描述

所有的针对互斥量的加锁解锁操作，都必须针对同一模块，并且在同一抽象层面进行。否则，将会可能导致某些加锁/解锁操作不会依照多线程设计而被执行，甚至依照锁的类型，最终导致死锁、资源竞争等其他漏洞的爆发。在对加锁对象进行解锁过程中，错误的解锁顺序，容易造成死锁。

漏洞与风险

死锁

不正确地加锁解锁操作，导致死锁、资源竞争等问题。

漏洞利用威胁

低

缓解与预防

正确地执行加锁解锁操作。对于死锁问题，可以采取例如可以调整资源请求顺序，或者加上sleep操作等方式。

数据竞争

在 2024-九月-28 发布于 并发和时序错误 分类 • 1 min read

  • 缺陷描述
  • 漏洞与风险
    • 不可预测的行为
  • 漏洞利用威胁
  • 缓解与预防
  • 演示实例
    • 漏洞代码示例
    • 漏洞代码修复示例
  • 引用

缺陷描述

满足如下条件时触发数据竞争

  • 一个进程中两个或多个线程同时访问同一个内存区域；
  • 至少一个访问是写操作；
  • 这些线程没有使用锁去控制对该内存区域的访问。

漏洞与风险

不可预测的行为

当数据竞争发生时，访问的顺序是不确定的，并且不同的运行依据不同的执行顺序，会给出不同的计算结果。虽然有些数据竞争是良性的（例如循环等待中的内存访问），但是很多数据竞争在程序中是缺陷。

漏洞利用威胁

低

缓解与预防

  • 使用合适的同步手段（例如锁，mutex等）。
  • 使用线程之前仔细考虑。

演示实例

漏洞代码示例

int var;

void Thread1() {  // 执行一个线程
  var++;
}
void Thread2() {  // 执行另一个线程
  var++;
}

这个最简单的数据竞争，同时也是最可能发生的数据竞争：两个线程都不经过任何的同步访问同一个变量。多数情况下这个数据竞争是无害的（允许数据不那么精确），但是有些是极端有害的（例如这个是你的银行账户）。

int var;

void Thread1() {  // 执行一个线程
  lock(threadMutex);
  var++;
  unlock(threadMutex);
}
void Thread2() {  // 执行另一个线程
  lock(threadMutex);
  var++;
  unlock(threadMutex);
}

在修复后的代码中，两个加锁和两个解锁操作被加入到程序中用于构成临界区，这个临界区用于保护变量var。
